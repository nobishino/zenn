## Go言語を完全に理解したい人が読むべきもの

Go言語を「完全に」理解したい人が読むべきドキュメントは、少なくとも2つあります。


| タイトル | 内容 | 長さ・難しさ |
| --- | --- | --- |
| [The Go Programming Language Specification](https://go.dev/ref/spec) | ・Go言語の文法<br>・単一goroutineの動作<br>・複数goroutine、つまり並行処理の動作については書いてない | ・長い<br>・読もうと思えば読める難しさ（※個人差があります） |
| [The Go Memory Model](https://go.dev/ref/mem) | ・並行処理の動作 | ・短い<br>・**予備知識がないととても読めないくらい難しい** |

1つめは [The Go Programming Language Specification](https://go.dev/ref/spec)、つまりGoの言語仕様書です。Go言語の文法や意味論のほとんどはここに書かれています。

しかし、言語仕様書だけではGoプログラムの振る舞いを完全には説明できません。そこで2つめのドキュメントとして [The Go Memory Model](https://go.dev/ref/mem) があります。これは、複数のgoroutineが絡む「並行処理」におけるプログラムの振る舞い、特に**並行するメモリーの読み書き**について規定するドキュメントです。

ところがこのThe Go Memory Modelは筆者の感覚では非常に行間が広く、専門知識のある読者向けに書かれているように見えます。これを独力で理解することはほとんどのプログラマにとってかなりの困難だと思います。実際、筆者自身も勉強会に参加してその参加者の方々に助けられつつなんとか一定の理解に到達しました。

本書はそのようにして埋めてきた「行間」を図解することにより、専門知識のない読者がGoメモリーモデルの基礎を習得する助けになりたいと思います。

## 本書の流れと得られるもの

- まず第2章で並行処理の難しさをクイズを通して体験し、私たちが普段無意識に使っているメンタルモデル（逐次一貫モデル）が破綻する例を見ます。この現象を仕様として記述できるためにはメモリーモデルが必要です。
- 次に第3章で、メモリーモデルの中心概念である「観測可能性」と「happens-before関係」を図解で正確に理解し、第2章のクイズを正しく(仕様に基づいて)解けるようになります。ここまで読めば本書を書いた目的は達成されているといっても構いません。
- 第4章では、happens-before関係をよりフォーマルに定義するために「sequenced before」と「synchronized before」という2つの関係を説明します。これにより5章の理解を準備します。
- 第5章では、The Go Memory Modelの「Synchronization」節に列挙されている個々の同期演算（初期化、goroutineの生成・終了、チャネル通信、ロック、Once）を、第4章までの知識で読み解きます。
- 第6章では、Go 1.19のメモリーモデルアップデートで記述が追加されたsync/atomicパッケージについて学びます。

読み終わる頃には、The Go Memory Modelを自力で読み解くための土台が手に入っているはずです。

## メモリーモデルは読まないといけないのか

先に断っておくと、通常の開発上は、メモリーモデルを全く読む必要がありません。むしろ、読む必要がない状態こそが望ましいと言えます。(それがなぜなのかは、第2章で判明します)

「読む必要がない状態」を保つには、並行プログラムのテストを実行するときに`-race`オプションをつけてData Race Detectorを有効にするのが役に立ちます。

本書ではData Race Detector自体には深入りしません。次の公式ドキュメントを参考にしてください。
参考: [Data Race Detector - The Go Programming Language](https://go.dev/doc/articles/race_detector)
