# レビューノート（2026-07-10 通読レビュー）

books/go-memory-model 全章の通読レビュー。重要度順。各項目は対応したらチェックを付ける。

**全体所感**: 2章のクイズ→破綻→3章で解決、という骨格は強力。4・5章の追加によって「仕様の言葉（sequenced/synchronized before）」と「グラフの描き方」が往復できる構成になった。引用＋拙訳＋図＋実行可能コードのリズムも確立している。

## 技術的に不正確な箇所

- [x] **1. 5章チャネル・ルール2の「置き換えただけ」が不正確** — `5.synchronization.md` L243「先ほどの例の`c <- 0`を`close(c)`に置き換えただけです」とあるが、ルール1の例は`make(chan int, 10)`、close例は`make(chan int)`でチャネルの容量も変わっている。原文のmem docは「前の例（バッファ10）のsendをcloseに置き換えても同じ保証」と言っているので、close例もバッファ10に揃えるのが正確。

## 不親切な記述（説明の欠落）

- [x] **2. 3章の「罠」アラートが警告しっぱなしで説明ゼロ** — `3.happens-before-relation.md` L45-49。「『先に起こる』と解釈しないでください！！！（重要）」と最大級の警告をしながら、なぜダメなのか・ではどう考えるべきなのかを本書のどこにも書いていない。「…」で文が宙吊り。答えの材料は同じ章にある：観測可能性の条件2により「並行な書き込みも観測できる」＝happens-beforeが無くても値は見えるし、`A < B`があっても「時間的にAが先に実行された」ことを意味しない。アラートを条件2の後で回収する一節を足す。
- [x] **3. 観測可能性・条件2に一言のコメントもない** — `3.happens-before-relation.md` L247-249。「rとwが並行なら観測できる」は本書全体で最も直観に反するルールなのに、仕様を掲示しただけで通過している。「並行＝どちらを読むかは決まらない。だから*どちらもあり得る*と扱う（これがdata raceの不定性の正体）」という一言が欲しい。
- [x] **4. 5章ルール4とセマフォの接続が説明されていない** — `5.synchronization.md` L321「カウンティングセマフォとして使うときの根拠になるルールです」と言うだけで、なぜこのルールが「同時に最大3つ」を導くのかが書かれていない。「4回目のsend（4個目のgoroutineの`limit <- 1`）は、1回目のreceiveが完了するまで完了できない。だからバッファ満杯の間、後続のgoroutineはそこでブロックする」の一段が必要。プログラム例の解説（L366）もルールを経由せず結論だけ述べている。
- [ ] **5. 5章goroutine終了の図が図として機能していない** — `5.synchronization.md` L154-158。孤立ノード2つが並ぶだけで情報がほぼゼロ。実際に`hello()`のグラフを描くと `[go文] → [a = "hello"]` と `[go文] → [print(a)]` という形になり、これは3章ルール2で導入したS→X/Y型（並行）そのもの。既習の図型と接続すれば「矢印が無い」ことが視覚的に納得できる。
- [ ] **6. チャネルの4ルールだけ拙訳がない** — 5章の他の節（初期化・goroutine生成/終了・ロック・Once）と6章は全て引用＋拙訳なのに、チャネルの4つの引用（L194ほか）には拙訳がない。特にルール4の "The kth receive on a channel with capacity C…" は訳が最も欲しい文。またルール2とルール4の図だけキャプションがない。
- [ ] **7. 2章がWaitGroupを無説明で使う** — `2.sequential-consistency.md` L18。想定読者が「Memory Modelは読んだことがないGopherくん」なら、WaitGroupの1行説明（カウンタが0になるまでWaitで待つ）はあってよい。さらにL125「このクイズで登場する演算は、次の4つです」で`wg.Done()`等を黙って除外している。「同期のための演算はいったん脇に置く」と断るべき（3章で「同期演算だけは矢印になる」という伏線にもなる）。
- [ ] **8. 3章のWaitGroup矢印の根拠が5章まで宙に浮く** — `3.happens-before-relation.md` L101-104で「メモリーモデルで決められた所定のペア」としてDone→Waitを描くが、5章で「WaitGroupは実はSynchronization節に登場しない」と明かされる。3章の時点で「個々の同期演算がどんな矢印を作るかの根拠は第5章で扱います」と前方参照を入れる。

## 発展的な提案（余力があれば）

- [ ] **18. 「保証される」と「できる」のギャップ** — 1章・3章冒頭の引用は「観測することが*保証される*条件」だが、3章で提示する仕様は「観測*できる*(may observe)」の条件で、両者は別物（保証には「他の値を観測し得ない」ことまで要る）。「本書は『あり得る観測』の側を扱う」と一言スコープ宣言すると誠実。
- [ ] **19. DRF-SCの回収** — 2章の「data raceのないプログラムは逐次一貫モデルで説明できる」という実践上最重要の主張が、道具が出揃った後も最後まで根拠を示されない。証明は不要だが、6章末か7章で直観の一段（「data raceが無い＝並行な読み書きの組が無い＝観測可能性の条件2が発動しない」）か参考文献への誘導があると本の主張が閉じる。
- [ ] **20. 2章Venn図のASCIIアート** — モバイルでは横スクロール・罫線ズレのリスクがある。図解が売りの本なので画像化を検討。
