## 第2章の登場人物紹介

![Gopherくん](/images/gopher.png =240x)
*Gopherくん（絵: 発表者）*

**Gopherくん**は、メモリーモデルを学ぶわたしたちの代表です。Goのプログラムを書くことは得意ですが、[The Go Memory Model](https://go.dev/ref/mem) は読んだことがありません。そのため、難しいクイズには間違った考えを述べることがあります。

## Quiz (Message Passing Test)

次のクイズを考えてみましょう。[https://go.dev/ref/mem#badsync](https://go.dev/ref/mem#badsync) を少し改変したものです。

```go
var a, b int
var wg sync.WaitGroup

func f() {
	defer wg.Done()
	a = 1
	b = 1
}

func g() {
	defer wg.Done()
	r1 := b
	r2 := a
	if r1 == 1 && r2 == 0 { // これは発生しうるか？
		panic("Answer: Yes") // 発生したらpanic
	}
}

func exec() {
	wg.Add(2)
	defer wg.Wait() // f, gの完了を待ち合わせる

	go f()
	go g()
}
```

プログラムの構成を確認しておきます。

- `exec`, `f`, `g` という3つの関数がある
  - `main`関数で`exec`が実行されるとしてください
- `exec`は`f`, `g`を別goroutineで起動して待ち合わせる
- `f`はグローバル変数`a`, `b`に書き込む
- `g`はグローバル変数`a`, `b`から読み出す

**問題: `(r1, r2) = (1, 0)`になりうるか？ つまり、このプログラムはpanicする可能性がありますか？**

### 実験による答え

実際に`exec`を無限ループで回して実験してみると、panicが発生します。つまり実験による正しい答えは**YES**です。

![実験結果: panic: Answer: Yes が発生している](/images/messagepassing-experiment.png)
*実験結果。`go run ./messagepassing` を実行すると `panic: Answer: Yes` が発生する*

## Gopherくんの考え(1)（※実は正しくありません）

さて、皆さんは実験結果を見たので、この問題の答えがYES、つまりpanicすることがあり得ることを知っています。ここでは一旦その答えを知らないものとして、Gopherくんと一緒にクイズを解いてみましょう。

Gopherくんは次のように考えました。

> 並行処理プログラムは難しい。なぜなら演算の順序が確定していないからね。でも小さなプログラムであれば、起こりうる全ての順序を列挙すれば完全に分析できるはず！

## Gopherくんの考え(2)

このクイズで登場する演算は、次の4つです。

- `f`のgoroutineで実行される演算: `a = 1`, `b = 1`
- `g`のgoroutineで実行される演算: `r1 := b`, `r2 := a`

Gopherくんは、「すべての演算を一列に並べる方法としてありうるパターンを全て列挙しよう。ただし、同一goroutineで行われる演算はプログラムに書かれている通りの順序で並べることにする」と考えました。

この4つの演算を一列に並べる方法は全部で6通りあるので、そのそれぞれの実行結果を予想すれば良いことになります。

| パターン | 実行順序 | 結果 `(r1, r2)` |
| --- | --- | --- |
| 1 | `a = 1` → `b = 1` → `r1 := b` → `r2 := a` | `(1, 1)` |
| 2 | `a = 1` → `r1 := b` → `b = 1` → `r2 := a` | `(0, 1)` |
| 3 | `a = 1` → `r1 := b` → `r2 := a` → `b = 1` | `(0, 1)` |
| 4 | `r1 := b` → `a = 1` → `b = 1` → `r2 := a` | `(0, 1)` |
| 5 | `r1 := b` → `a = 1` → `r2 := a` → `b = 1` | `(0, 1)` |
| 6 | `r1 := b` → `r2 := a` → `a = 1` → `b = 1` | `(0, 0)` |

たとえばパターン1は、`f`の演算2つが`g`の演算2つよりも先に実行されるパターンです。両方の書き込みが読み込みより先に完了するので、`(r1, r2) = (1, 1)`となります。

パターン2は、`b`からの読み出しが`b`への書き込みより先に行われるパターンです。ゼロ値を読み込むことになるので`r1 = 0`となります。

同様に6パターン全ての結果を考えると、

- ありうるのは`(1,1)`, `(0,1)`, `(0,0)`の3パターン
- `(1,0)`になるパターンはひとつもなかった

ということがわかります。

## Gopherくんの考え(結論)

Gopherくんの結論は次の通りです。

> - 全ての演算を並び替える方法は、6通りある（※同一goroutineで行われる演算の順序を変えないものに限る）
> - この6通りのどれを考えても、`(r1, r2) = (1, 0)`となることはない
> - ゆえに、Quiz (Message Passing Test)の答えは "No" である

**不正解！ 答えはYesです。** 実験をすると、`(r1, r2) = (1, 0)`となってpanicする場合があるのでした。

## Gopherくんの考えはどこが間違っているのか

実験の結果から、Message Passing Testの正しい答えは "Yes" であることがわかりました。これはちょっと考えると不思議に思えます。Gopherくんの考えた筋道は非常に理にかなっていてもっともらしいものだったからです。Gopherくんの考えはどこが間違っていたのでしょうか？ それに答えるために、私たちがいつも無意識に使っているメンタルモデルそのものに実は間違いがある、という話をこれからしていきます。

実は、「すべての演算を何らかの方法で一列に並べて、その順序で演算が実行されるものと考える」というメンタルモデルは正しくない場合があります。GopherくんがMessage Passing Testに適用した考え方は、実は無意識にこのメンタルモデルを使っていたのです。それを思い出してみましょう。

Gopherくんは、4つの演算を1列に並べる方法を6通り列挙して、その6通りそれぞれに対してプログラムの出力を考えるという方法で問題を解こうとしました。どんな順序で演算が実行されるかはわからないけれども、ともかく何らかの順序で演算が実行されるのだから、それを全て列挙すればこのプログラムを理解できるだろう、と考えたわけでした。

しかしこの考え方では、暗黙のうちに仮定してしまっていることがあります。それは、**「プログラムの挙動は全ての演算を一列に並べた何らかの順序に従って1つずつ実行されたものとして説明できる」という仮定**です。この仮定は、学術的には**逐次一貫モデル**というものに相当する仮定です。この仮定は、多くのプログラムにおいては成り立つのですが、一部のプログラムにおいては成り立ちません。そして、Message Passing Testは、「逐次一貫モデル」が成り立たない一部のプログラムの具体例なのです。

## 正しく解くためにはメモリーモデルが必要である

それでは、このクイズを正しく解くにはどうすれば良いのでしょうか？ メモリーモデルの正しい知識があれば解けるようになります。その知識は次の章で身につきますが、その前に、このクイズが示唆していることを振り返って考察しておきましょう。

## 「並行処理の難しさ」にも2つのレベルがある

この章で扱ったクイズのプログラムは、「並行処理の難しさ」とは何なのか、についての私たちの理解を一段階深めてくれます。それは、「並行処理の難しさ」というものに2つのレベルがあるということです。

「並行処理が難しい」ということそれ自体は、経験を積んだプログラマーなら多かれ少なかれ知っていることだと思います。しかし、その難しさには、少なくとも2つのレベルがあります。それを表したのが次のVenn図です。このVenn図は、この世に存在しうるあらゆるGo言語のプログラムを分類しているものです。

```
┌───────────────────────────────────────────────────┐
│ 並行性があり、逐次一貫モデルで説明できないプログラム      │
│                                                   │
│   ┌───────────────────────────────────────────┐   │
│   │ 並行性があり、逐次一貫モデルで説明できるプログラム │   │
│   │                                           │   │
│   │   ┌───────────────────────────┐           │   │
│   │   │   並行性のないプログラム       │           │   │
│   │   └───────────────────────────┘           │   │
│   └───────────────────────────────────────────┘   │
└───────────────────────────────────────────────────┘
```

### 並行性のないプログラム

まず真ん中には、並行性(concurrency)の存在しないGoプログラムがあります。大雑把に言えば、main goroutineだけを使って書いているプログラムだと理解してください。これらが、最も理解しやすいプログラムだと言えるでしょう。

### 並行性があり、逐次一貫モデルで説明できるプログラム

次にその外側に、並行性(concurrency)が存在するが、逐次一貫モデルで説明可能なプログラムがあります。

これらのプログラムは、いろいろな演算の順序が確定していないので、可能な実行結果のパターンが非常に大きくなる場合があります。そのため理解するのが難しい場合が多いです。[組合せ爆発](https://ja.wikipedia.org/wiki/%E7%B5%84%E5%90%88%E3%81%9B%E7%88%86%E7%99%BA)による難しさだ、と解釈しても良いでしょう。

しかしそれでも、このグループのプログラムは、あくまで全ての演算が一列に順序よく実行されたものとして分析できます。つまり、「並行プログラムの中では比較的簡単に理解できる並行プログラム」だとも言えるのです。

### 並行性があり、逐次一貫モデルで説明できないプログラム

最後に、このさらに外側には、「並行性が存在して、しかも、逐次一貫モデルで説明不可能なプログラム」があります。

これらはもっとも理解するのが困難なプログラムだと言えます。なぜなら、演算の順序が確定していないどころか、そもそも演算が一列に順序よく行われるという考え自体が成り立たない世界にあるプログラムだからです。先ほどのMessage Passing Testがこのグループのプログラムの具体例です。なお、このようなプログラムは（書こうと思えば書けますが）書くべきではありません。

そして、この全体像を掴んだことで、「なぜメモリーモデルは難しいのか」という疑問にも答えることができます。それは、メモリーモデルというものは、このVenn図の最も外側にある、最も理解が難しいプログラム、つまり逐次一貫モデルが成り立たないようなプログラムも含めて、全てのGoプログラムの挙動を説明するものだからです。それは、言い換えれば、「全ての演算が一列に順序よく実行されたものとして分析できないプログラム」の挙動を説明するという難しい課題があるということです。この難しい課題を解決するためには、どうしてもいくらか大掛かりな枠組みが必要となります。それが、メモリーモデルが難しい理由です。

## 2つのレベルを分ける条件は "data race" の有無

この2レベルの並行プログラムの境界線を決める条件について、少し先取りして説明します。

実は、Goメモリーモデルを注意深く検討すると、**data raceのないプログラムはこのVenn図の内側の2つの部分集合の中に必ず入る**ことが保証されます。つまり、data raceのないプログラムは、逐次一貫モデルで説明できるゾーンに入るのです。だからこそ、Race Detectorでテストすると良いのです。

ここで、data raceというものが何なのかはまだ説明していませんが、とりあえずは「メモリーの読み書きが競合すること」くらいに理解していただければ大丈夫です。

## この章のまとめ

- 「プログラムの実行結果は、全ての演算を一列に並べてその順序で演算を実行した結果として説明できる」という考えを**逐次一貫モデル**という
- 「逐次一貫モデル」が成り立たないGoプログラムが存在する
- data raceのないプログラムに対しては、「逐次一貫モデル」が成り立つ
- The Go Memory Modelは、逐次一貫モデルの成り立たないプログラムも含めた全てのGoプログラムの振る舞いを説明する理論であり、それゆえに難しい
- Message Passing Testを理解するためには、正確なメモリーモデルが必要である
