## 本書のまとめ

**第2章 並行処理の難しさと逐次一貫モデルの破綻**

- 「プログラムの実行結果は、全ての演算を一列に並べてその順序で演算を実行した結果として説明できる」という考えを「逐次一貫モデル」という
- 「逐次一貫モデル」が成り立たないGoプログラムが存在する（Message Passing Testがその具体例）
- data raceのないプログラムに対しては、「逐次一貫モデル」が成り立つ
- The Go Memory Modelは、逐次一貫モデルの成り立たないプログラムも含めた全てのGoプログラムの振る舞いを説明する理論であり、それゆえに難しい

**第3章 観測可能性とhappens-before関係**

- メモリーモデルのゴールは「観測可能性」を定めることであり、happens-before関係はそのための概念的道具である
- happens-before関係はグラフを描いて求められる
  - 同一goroutineの演算はプログラム順に矢印で繋ぐ
  - 異なるgoroutine間は同期演算のペアがあるときに矢印で繋ぐ
- グラフ上でAからBに到達できるとき`A < B`（A happens before B）、お互いに到達できないときAとBは並行(concurrent)
- 読み込み演算`r`が書き込み演算`w`を観測できるのは、(1) `w < r`かつ間に他の書き込みが挟まらないとき、または (2) `r`と`w`が並行であるとき

**第4章 happens beforeの正体：sequenced beforeとsynchronized before**

- The Go Memory Modelは、happens beforeを直接定義するのではなく、sequenced beforeとsynchronized beforeという2つの関係の推移的閉包として定義している
- sequenced beforeは同一goroutine内の演算順序、synchronized beforeは同期演算同士の順序に対応する
- 第3章のhappens-beforeグラフの「第二段階」はsequenced beforeを、「第三段階」はsynchronized beforeを、それぞれ矢印として描いていた

**第5章 同期演算(Synchronization)を読み解く**

- The Go Memory Modelの「Synchronization」節は、Go言語の各同期の仕組みがどんなsynchronized before関係(矢印)を作るかを規定している
- 初期化・goroutineの生成・goroutineの終了・チャネル通信・ロック・Onceについて、それぞれの同期演算がどこにどんな矢印を引くのかを確認した
- goroutineの終了それ自体は他の演算とhappens-before関係を持たないため、効果を他のgoroutineに観測させるには明示的な同期が必要である

**第6章 Go1.19メモリーモデルとsync/atomicパッケージ**

- Go1.19でメモリーモデルが8年ぶりにアップデートされ、sync/atomicについての記述が追加された
- sync/atomicだけで書かれたGoプログラムは逐次一貫モデルで説明できる
- sync/atomicは「分割できない」だけでなく、happens-before関係を作り出す同期演算としての性質も持つ
